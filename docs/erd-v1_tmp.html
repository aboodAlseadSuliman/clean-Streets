CleanStreets ERD v1

الفكرة العامة

المشروع لا يجب أن يبنى حول "الطلب" فقط، بل حول "الحالة" أيضًا.

السبب: قد يرسل أكثر من مواطن بلاغًا عن نفس السيارة المدمرة، لذلك من الأفضل فصل:

  • reports: البلاغات الواردة من المواطنين أو الجهات المحلية.
  • vehicle_cases: الحالة الفعلية للسيارة في الشارع بعد دمج البلاغات المتكررة.

هذا التصميم مناسب جدًا مع Laravel + Filament لأن:

  • Filament يدير لوحة الموظفين والجهات بسهولة.
  • يمكن لاحقًا إضافة workflow واضح للحالات.
  • يمكن تتبع كل تغيير على الحالة بدون ضياع أصل البلاغ.

الكيانات الأساسية

1. users

مستخدمو النظام الداخليون فقط، مثل:

  • مدير النظام
  • موظف استقبال البلاغات
  • مراقب ميداني
  • موظف الجهة المنفذة

أهم الحقول:

  • id
  • name
  • email nullable
  • phone
  • password
  • agency_id nullable
  • role أو يدار لاحقًا عبر permissions package
  • is_active
  • timestamps

2. agencies

تمثل الجهات المعنية، مثل:

  • مجلس المدينة
  • مديرية النقل
  • فرع المرور
  • الشركة السورية للمعادن
  • لجنة حي

أهم الحقول:

  • id
  • name
  • type (municipality, traffic, transport, recycling, district_committee, other)
  • phone nullable
  • address nullable
  • is_active
  • timestamps

3. districts

اختياري لكنه مهم إذا أردنا التقارير على مستوى المدينة.

أهم الحقول:

  • id
  • name
  • code nullable
  • timestamps

4. neighborhoods

الأحياء ضمن المدينة.

أهم الحقول:

  • id
  • district_id nullable
  • name
  • code nullable
  • timestamps

5. reporters

الجهة أو الشخص الذي أرسل البلاغ. يفضل فصلها عن users لأن أغلب المبلّغين لن يملكوا حسابات داخل النظام.

أهم الحقول:

  • id
  • full_name
  • phone
  • whatsapp_phone nullable
  • source_type (citizen, district_committee, agency_staff, anonymous)
  • notes nullable
  • timestamps

6. vehicle_cases

هذا هو السجل الرئيسي لكل سيارة مدمرة أو هيكل سيارة في الشارع.

أهم الحقول:

  • id
  • case_number unique
  • neighborhood_id
  • location_text
  • nearest_landmark nullable
  • latitude nullable
  • longitude nullable
  • vehicle_type (sedan, pickup, truck, bus, motorcycle, unknown)
  • damage_type (burned, destroyed, abandoned_wreck, partially_damaged, unknown)
  • plate_number nullable
  • color nullable
  • make_model nullable
  • priority (low, medium, high, critical)
  • current_status (new, under_review, verified, duplicate, assigned, scheduled, removed, rejected, closed)
  • reported_reports_count default 0
  • first_reported_at
  • last_reported_at
  • verified_at nullable
  • removed_at nullable
  • created_by nullable
  • timestamps

ملاحظات:

  • هذا الجدول يمثل "الحقيقة التشغيلية" داخل النظام.
  • يمكن دمج عدة بلاغات داخله.

7. reports

كل بلاغ وارد من مواطن أو لجنة أو جهة.

أهم الحقول:

  • id
  • reference_number unique
  • vehicle_case_id nullable
  • reporter_id nullable
  • submitted_by_user_id nullable
  • channel (web, whatsapp, phone, field_entry, district_committee, agency)
  • neighborhood_id nullable
  • location_text
  • nearest_landmark nullable
  • latitude nullable
  • longitude nullable
  • description nullable
  • vehicle_type nullable
  • damage_type nullable
  • plate_number nullable
  • status (pending_review, linked_to_case, duplicate, rejected)
  • review_note nullable
  • submitted_at
  • reviewed_at nullable
  • timestamps

ملاحظات:

  • يمكن أن يبقى البلاغ بدون ربط بحالة حتى يراجعه الموظف.
  • بعد المراجعة يربط مع vehicle_cases أو يرفض أو يعتبر مكررًا.

8. report_attachments

صور البلاغات والمرفقات الأولية.

أهم الحقول:

  • id
  • report_id
  • file_path
  • file_name
  • mime_type
  • file_size nullable
  • caption nullable
  • timestamps

9. inspections

التحقق الميداني أو الإداري من وجود السيارة وحالتها الفعلية.

أهم الحقول:

  • id
  • vehicle_case_id
  • inspector_user_id
  • agency_id nullable
  • inspection_result (confirmed, not_found, duplicate, out_of_scope, deferred)
  • notes nullable
  • inspected_at
  • timestamps

10. case_assignments

إحالة الحالة إلى جهة معينة أو موظف محدد.

أهم الحقول:

  • id
  • vehicle_case_id
  • agency_id
  • assigned_to_user_id nullable
  • assigned_by_user_id nullable
  • assignment_status (active, accepted, in_progress, completed, cancelled)
  • assigned_at
  • due_at nullable
  • notes nullable
  • timestamps

11. removals

تسجيل عملية الإزالة نفسها عند التنفيذ.

أهم الحقول:

  • id
  • vehicle_case_id
  • assignment_id nullable
  • agency_id
  • executed_by_user_id nullable
  • scheduled_at nullable
  • started_at nullable
  • completed_at nullable
  • result (removed, partially_removed, failed, cancelled)
  • destination nullable
  • notes nullable
  • timestamps

12. case_status_logs

سجل زمني لكل انتقال حالة.

أهم الحقول:

  • id
  • vehicle_case_id
  • from_status nullable
  • to_status
  • changed_by_user_id nullable
  • reason nullable
  • created_at

العلاقات

  • agencies 1:N users
  • districts 1:N neighborhoods
  • neighborhoods 1:N vehicle_cases
  • neighborhoods 1:N reports
  • reporters 1:N reports
  • vehicle_cases 1:N reports
  • reports 1:N report_attachments
  • vehicle_cases 1:N inspections
  • vehicle_cases 1:N case_assignments
  • vehicle_cases 1:N case_status_logs
  • vehicle_cases 1:N removals
  • agencies 1:N case_assignments
  • agencies 1:N removals

تسلسل العمل المقترح

  1. المواطن يرسل بلاغًا مع صورة وموقع تقريبي.
  2. الموظف يراجع البلاغ.
  3. إذا كانت السيارة جديدة ينشأ vehicle_case.
  4. إذا كانت مكررة يربط البلاغ مع حالة موجودة.
  5. تسجل زيارة تحقق في inspections.
  6. تحال الحالة إلى جهة عبر case_assignments.
  7. عند التنفيذ تسجل العملية في removals.
  8. كل انتقال حالة يسجل في case_status_logs.

ملاحظات مهمة قبل التنفيذ

  • لا أنصح بجعل المواطن مستخدمًا داخليًا في users في المرحلة الأولى.
  • لا أنصح بالاكتفاء بجدول reports فقط لأن التكرار سيكون عاليًا جدًا.
  • إذا أردنا لوحة Filament قوية، فهذا النموذج مناسب جدًا للـ Resources والـ Relation Managers.
  • يمكن لاحقًا إضافة جدول case_comments للملاحظات الداخلية إذا احتجناه.
  • يمكن لاحقًا استبدال report_attachments بحل Media Library إن أردنا.
